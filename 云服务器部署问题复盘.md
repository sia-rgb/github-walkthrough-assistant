# 云服务器部署踩坑复盘：从“代码能跑”到“服务可用”

> 目标：记录真实的云端部署排障过程，在完成项目上线基础上，系统理解公网入口、进程托管、端口转发、静态资源路径和代码同步之间的关系。  
> 复盘围绕 **现象 → 判断 → 根因 → 解决 → 经验** 进行整理，方便后续复用。

---

## 1. 部署链路总览

这次部署涉及的不只是 Python 项目本身，而是一条完整的 Web 服务链路：

```text
浏览器公网访问
↓
Nginx 接收 HTTP 请求
↓
Nginx 按路径转发到本机端口
↓
127.0.0.1:8000 / 127.0.0.1:8010
↓
systemd 托管 Python Web 服务
↓
项目代码处理请求并返回页面 / API
```

最终形成了两个项目共存的部署结构：

```text
http://服务器IP/
→ Nginx
→ 127.0.0.1:8000
→ economist-content-pipeline / ecno-assist

http://服务器IP/github-walkthrough-assistant/
→ Nginx
→ 127.0.0.1:8010
→ github-walkthrough-assistant
```

核心理解：

```text
代码能手动跑起来 ≠ 服务已经稳定部署
```

真正的部署要同时满足：

```text
代码版本正确
依赖环境正确
后端进程长期运行
端口监听正常
Nginx 转发路径正确
静态资源能正常加载
服务重启后可自动恢复
```

## 2. 问题分类总览

| 类别 | 典型问题 | 本质原因 | 关键经验 |
|---|---|---|---|
| 后端进程 | 502 Bad Gateway | Nginx 找不到可用后端服务 | 先查端口和 systemd，不要盲目重装 |
| 进程托管 | 本地电脑重启后服务失效 | 后端进程依赖 SSH 会话 | 生产部署必须使用 systemd |
| 启动入口 | Uvicorn 找不到 app 模块 | 项目入口路径判断错误 | 不猜入口，先查 README / 启动脚本 |
| Python 环境 | `.venv` 不存在 | GitHub 不会提交虚拟环境 | 服务器上重新创建独立虚拟环境 |
| 代码同步 | GitHub 更新后服务器没更新 | 服务器代码需要主动 pull | 本地开发，GitHub 管版本，服务器只部署 |
| 项目类型 | http.server 项目不能用 Uvicorn 启动 | Web 框架类型不同 | 先判断项目类型，再选启动方式 |
| systemd | `status=203/EXEC` | ExecStart 路径或命令错误 | systemd 中写完整 Python 路径 |
| 环境变量 | `.env` 写了但服务读不到 | systemd 不会自动读取项目 `.env` | 用 `EnvironmentFile` 显式加载 |
| Nginx 配置 | 改了 default 但不生效 | 实际加载的是其他站点配置 | 用 `nginx -T` 和 `grep` 查真实配置 |
| 路径转发 | 子路径访问返回 Not Found | Nginx 转发时保留了错误前缀 | 注意 `proxy_pass` 末尾 `/` |
| 静态资源 | 页面能打开但没样式 | CSS / JS 路径仍指向根路径 | 子路径部署必须处理 static/base path |
| 静态资源 | `/styles.css`、`/app.js` 返回 404 | FastAPI 只返回 index.html，未暴露 frontend 静态目录 | 使用 StaticFiles 挂载 `/static`，并同步修改 HTML 引用路径 |
| API 路径 | 点击“分析”后 `/api/analyze` 404 | 项目部署在 `/github-walkthrough-assistant/` 子路径，但前端 fetch 写成根路径 | 子路径部署时前端使用相对路径，如 `fetch("api/analyze")` |
| 长任务超时 | 路径修复后出现 504 Gateway Timeout | `/api/analyze` 同步执行完整分析，耗时约 110 秒，超过 Nginx 默认等待时间 | 调大 `proxy_read_timeout`，并考虑异步任务化 |

# 3. 后端服务类问题

## 3.1 502 Bad Gateway：Nginx 活着，但后端没响应

### 现象

浏览器访问公网地址时出现：

```text
502 Bad Gateway
```

一开始容易误判为：

```text
服务器坏了
公网 IP 有问题
Nginx 配置坏了
GitHub 项目没有同步成功
```

### 排查

先确认后端进程是否存在：

```bash
ps aux | grep uvicorn
```

如果只看到：

```bash
grep --color=auto uvicorn
```

说明没有真实的 Uvicorn 进程。

再看端口是否监听：

```bash
ss -lntp | grep 8000
```

如果没有任何输出，说明：

```text
Nginx 正在转发到 127.0.0.1:8000
但 8000 端口根本没有服务在跑
```

### 根因

502 的直接含义是：

```text
Nginx 收到了公网请求
但它转发到后端端口时，后端没有响应
```

也就是说，这不是“公网入口”问题，而是“后端服务不可用”问题。

### 解决方式

先手动启动后端验证：

```bash
cd /root/economist-content-pipeline
source .venv/bin/activate
python -m uvicorn app:APP --app-dir src --host 127.0.0.1 --port 8000
```

确认能访问后，再交给 systemd 托管：

```ini
[Unit]
Description=Economist Content Pipeline FastAPI Service
After=network.target

[Service]
User=root
WorkingDirectory=/root/economist-content-pipeline
ExecStart=/root/economist-content-pipeline/.venv/bin/python -m uvicorn app:APP --app-dir src --host 127.0.0.1 --port 8000
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

启动并检查：

```bash
sudo systemctl daemon-reload
sudo systemctl enable economist-content-pipeline
sudo systemctl start economist-content-pipeline
sudo systemctl status economist-content-pipeline
ss -lntp | grep 8000
```

### 经验

遇到 502 时，不要直接重装环境。优先按顺序检查：

```text
Nginx 是否正常
↓
后端端口是否监听
↓
systemd 服务是否 running
↓
后端日志是否报错
```

## 3.2 本地电脑重启后，云端服务失效

### 现象

本地电脑重启后，公网地址再次变成 502。

直觉上会觉得奇怪：

```text
云服务器不是独立运行的吗？
为什么本地电脑重启会影响云服务器？
```

### 根因

不是云服务器依赖本地电脑，而是之前的后端进程依赖 SSH 会话。

如果通过 SSH 登录服务器后，在终端里直接执行：

```bash
uvicorn ...
```

这个进程可能会随着 SSH 断开而退出。

### 解决方式

使用 systemd 托管服务，而不是依赖手动命令。

systemd 解决的是：

```text
SSH 断开后服务仍然运行
服务器重启后服务自动启动
服务异常退出后自动重启
通过统一命令查看状态和日志
```

### 经验

手动启动只适合验证，不适合作为部署方式。

判断服务是否真正部署成功，要看：

```bash
sudo systemctl status 服务名
ss -lntp | grep 端口号
```

只要服务是 `active (running)`，端口也在监听，才算完成稳定部署。

# 4. Python 项目启动类问题

## 4.1 Uvicorn 启动入口写错

### 现象

执行：

```bash
uvicorn app:APP --host 127.0.0.1 --port 8000
```

报错：

```text
Error loading ASGI app. Could not import module "app".
```

改成：

```bash
uvicorn app:app --host 127.0.0.1 --port 8000
```

仍然失败。

### 根因

项目根目录下没有 `app.py`，所以 `app:APP` 找不到。

后面通过搜索启动命令：

```bash
grep -R "uvicorn" -n .
```

发现真实启动方式是：

```bash
python -m uvicorn app:APP --app-dir src --host 127.0.0.1 --port 8000
```

也就是说，入口文件在 `src` 目录下。

### 解决方式

使用正确命令：

```bash
python -m uvicorn app:APP --app-dir src --host 127.0.0.1 --port 8000
```

其中：

```text
--app-dir src
```

表示让 Uvicorn 到 `src` 目录中寻找 `app.py`。

### 经验

部署 FastAPI 项目时，不要凭感觉猜入口。优先查：

```bash
grep -R "uvicorn" -n .
grep -R "FastAPI" -n .
```

常见启动形式包括：

```bash
python -m uvicorn app:APP --app-dir src --host 127.0.0.1 --port 8000
python -m uvicorn main:app --host 127.0.0.1 --port 8000
python -m uvicorn src.main:app --host 127.0.0.1 --port 8000
```

核心规则是：

```text
模块路径:FastAPI对象名
```

例如：

```text
src/app.py 中定义 APP = FastAPI()
→ app:APP --app-dir src

src/main.py 中定义 app = FastAPI()
→ src.main:app
```

## 4.2 虚拟环境不存在

### 现象

执行：

```bash
source .venv/bin/activate
```

报错：

```text
No such file or directory
```

### 根因

GitHub 仓库通常不会提交 `.venv`。

服务器 clone 下来的只是代码和依赖声明，例如：

```text
requirements.txt
pyproject.toml
```

但不包含本地虚拟环境。

### 解决方式

在服务器上重新创建虚拟环境：

```bash
cd /root/economist-content-pipeline
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

如果系统缺少 venv 模块：

```bash
sudo apt update
sudo apt install -y python3-venv
python3 -m venv .venv
```

### 经验

每个项目应该有自己的虚拟环境：

```text
/root/project-a/.venv
/root/project-b/.venv
```

不要多个项目共用一个全局 Python 环境，否则依赖版本容易互相污染。

## 4.3 第二个项目不是 FastAPI，而是 http.server

### 现象

部署 `github-walkthrough-assistant` 时发现：

```text
项目中没有定义 FastAPI
主要 Web 服务逻辑在 src/web_app.py
使用的是 Python 标准库 http.server
```

### 根因

它不是 FastAPI 项目，所以不能直接用：

```bash
python -m uvicorn app:APP --app-dir src --host 127.0.0.1 --port 8010
```

### 可选方案

#### 方案一：继续使用 http.server

```ini
ExecStart=/root/github-walkthrough-assistant/.venv/bin/python /root/github-walkthrough-assistant/src/web_app.py
```

优点是改动少。  
缺点是扩展性较弱，不利于后续做 API、异步任务、上传、模型调用等能力。

#### 方案二：迁移到 FastAPI + Uvicorn

新增 `src/app.py`：

```python
from pathlib import Path

from fastapi import FastAPI
from fastapi.responses import FileResponse
from fastapi.staticfiles import StaticFiles

BASE_DIR = Path(__file__).resolve().parent.parent
FRONTEND_DIR = BASE_DIR / "frontend"

APP = FastAPI(title="GitHub Walkthrough Assistant")

APP.mount("/static", StaticFiles(directory=FRONTEND_DIR), name="static")


@APP.get("/")
def index():
    return FileResponse(FRONTEND_DIR / "index.html")
```

然后用 Uvicorn 启动：

```bash
python -m uvicorn app:APP --app-dir src --host 127.0.0.1 --port 8010
```

### 经验

部署前先判断项目类型：

```text
FastAPI 项目 → uvicorn
Flask 项目 → flask / gunicorn
标准库 http.server → python xxx.py
纯前端静态项目 → Nginx static root 或 FastAPI StaticFiles
```

不要默认所有 Python Web 项目都用 Uvicorn。

# 5. systemd 托管类问题

## 5.1 systemd 报错 status=203/EXEC

### 现象

查看服务状态：

```bash
sudo systemctl status github-walkthrough-assistant
```

出现：

```text
Active: activating (auto-restart)
Main PID: xxxx (code=exited, status=203/EXEC)
```

### 根因

`203/EXEC` 表示：

```text
systemd 的 ExecStart 命令无法执行
```

常见原因包括：

```text
Python 路径写错
.venv/bin/python 不存在
项目目录写错
命令参数写错
文件没有执行权限
```

这次问题集中在 `ExecStart` 写法不正确。

### 解决方式

修改服务文件：

```bash
sudo nano /etc/systemd/system/github-walkthrough-assistant.service
```

使用完整路径：

```ini
ExecStart=/root/github-walkthrough-assistant/.venv/bin/python -m uvicorn app:APP --app-dir src --host 127.0.0.1 --port 8010
```

然后重新加载并重启：

```bash
sudo systemctl daemon-reload
sudo systemctl restart github-walkthrough-assistant
sudo systemctl status github-walkthrough-assistant
ss -lntp | grep 8010
```

### 经验

systemd 里的命令不要写成：

```bash
python ...
```

而要写完整路径：

```bash
/root/项目名/.venv/bin/python ...
```

因为 systemd 的运行环境和当前 shell 不一样，它不知道你是否已经 `source .venv/bin/activate`。

## 5.2 `.env` 与 systemd 的关系

### 现象

`.env` 已经写了环境变量，但服务运行时可能仍然读不到。

### 根因

`.env` 只是保存配置的文件。  
systemd 不会因为项目目录里有 `.env` 就自动读取它。

完整链路应该是：

```text
.env 保存变量
↓
systemd 通过 EnvironmentFile 读取变量
↓
Python 进程拿到环境变量
↓
代码通过 os.getenv() 使用变量
```

### 推荐写法

在项目目录下创建 `.env`：

```bash
cd /root/github-walkthrough-assistant
nano .env
```

示例：

```env
OPENAI_API_KEY=your_api_key
ACCESS_PASSWORD=your_password
MODEL_NAME=gpt-5.5
```

设置权限：

```bash
chmod 600 .env
```

在 systemd 中加入：

```ini
EnvironmentFile=-/root/github-walkthrough-assistant/.env
```

其中前面的 `-` 表示：

```text
.env 文件不存在时，不要让服务启动失败
```

### 经验

不要长期依赖：

```bash
export XXX=xxx
```

因为它只在当前终端会话中有效。

更适合部署环境的方式是：

```text
.env + systemd EnvironmentFile
```

# 6. Nginx 转发类问题

## 6.1 改了 Nginx default，但配置不生效

### 现象

一开始编辑：

```bash
sudo nano /etc/nginx/sites-available/default
```

但新增配置没有生效。

### 根因

Nginx 实际读取的不是 `default`，而是另一个站点配置：

```text
/etc/nginx/sites-available/ecno-assist
/etc/nginx/sites-enabled/ecno-assist
```

通过搜索端口定位：

```bash
sudo grep -R "127.0.0.1:8000" -n /etc/nginx
```

发现真正配置在：

```text
/etc/nginx/sites-available/ecno-assist
/etc/nginx/sites-enabled/ecno-assist
```

### 解决方式

编辑真实的配置源文件：

```bash
sudo nano /etc/nginx/sites-available/ecno-assist
```

在同一个 `server {}` 中增加第二个项目的路径转发：

```nginx
location /github-walkthrough-assistant/ {
    proxy_pass http://127.0.0.1:8010/;
    proxy_http_version 1.1;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}
```

检查并重载：

```bash
sudo nginx -t
sudo systemctl reload nginx
```

### 经验

不要凭经验猜 Nginx 配置文件。

优先使用：

```bash
sudo grep -R "端口号" -n /etc/nginx
sudo nginx -T 2>&1 | grep -n -C 8 "关键词"
```

Nginx 的真实加载链路通常是：

```text
/etc/nginx/nginx.conf
↓
include /etc/nginx/conf.d/*.conf
include /etc/nginx/sites-enabled/*
↓
具体站点配置
```

## 6.2 sites-available 和 sites-enabled 的软链接关系

### 现象

发现：

```text
/etc/nginx/sites-enabled/ecno-assist
```

是一个软链接，于是产生疑问：

```text
是否要给 8010 单独建一个软链接？
是否应该删除软链接，统一写到 default？
```

### 根因

这是 Nginx 的常见组织方式：

```text
sites-available：存放可用配置源文件
sites-enabled：存放启用配置的软链接
```

也就是说：

```text
真正编辑：/etc/nginx/sites-available/ecno-assist
实际加载：/etc/nginx/sites-enabled/ecno-assist
```

### 解决方式

不需要删除软链接，也不需要为 8010 单独建软链接。

因为只有一个公网 IP，更适合在同一个 `server` 中用不同路径分流：

```nginx
server {
    listen 80;
    server_name 服务器IP;

    location / {
        proxy_pass http://127.0.0.1:8000;
    }

    location /github-walkthrough-assistant/ {
        proxy_pass http://127.0.0.1:8010/;
    }
}
```

### 经验

没有域名时，一个公网 IP 可以用：

```text
一个 server
多个 location
不同路径转发到不同本机端口
```

例如：

```text
/                              → 8000
/github-walkthrough-assistant/ → 8010
```

## 6.3 子路径访问返回 `{"detail":"Not Found"}`

### 现象

执行：

```bash
curl http://127.0.0.1/github-walkthrough-assistant/
```

返回：

```json
{"detail":"Not Found"}
```

但直接访问后端：

```bash
curl -i http://127.0.0.1:8010/
```

可以返回网页内容。

### 根因

这说明 Nginx 已经成功转发到了后端，但后端收到的路径不对。

后端认识：

```text
/
```

但不认识：

```text
/github-walkthrough-assistant/
```

如果 Nginx 写成：

```nginx
proxy_pass http://127.0.0.1:8010;
```

后端收到的仍然是：

```text
/github-walkthrough-assistant/
```

因此返回 404。

### 解决方式

Nginx 中写成：

```nginx
location /github-walkthrough-assistant/ {
    proxy_pass http://127.0.0.1:8010/;
}
```

注意 `8010` 后面的 `/`。

这个写法会把 location 前缀剥掉：

```text
公网 /github-walkthrough-assistant/
→ 后端 /
```

### 经验

`proxy_pass` 末尾的 `/` 很关键：

```nginx
proxy_pass http://127.0.0.1:8010/;
```

表示剥离 location 前缀。

```nginx
proxy_pass http://127.0.0.1:8010;
```

表示保留原始路径。

子路径部署时，这个差异会直接决定是正常访问还是后端 404。

# 7. 前端静态资源类问题

## 7.1 页面能打开，但没有样式

### 现象

浏览器能打开：

```text
http://服务器IP/github-walkthrough-assistant/
```

但页面像裸 HTML，没有 CSS 样式和交互效果。

### 根因

这说明：

```text
HTML 加载成功
CSS / JS / 图片等静态资源没有正确加载
```

常见原因是：

```text
项目原来假设自己部署在根路径 /
但实际部署在子路径 /github-walkthrough-assistant/
```

例如前端写的是：

```html
<link rel="stylesheet" href="/static/style.css">
<script src="/static/app.js"></script>
```

浏览器会请求：

```text
http://服务器IP/static/style.css
```

但第二个项目真实路径应该是：

```text
http://服务器IP/github-walkthrough-assistant/static/style.css
```

### 解决方式

让前端和后端都适配子路径部署。

目标路径关系应该是：

```text
页面：
/github-walkthrough-assistant/

静态资源：
/github-walkthrough-assistant/static/...
```

可以使用 FastAPI 的 `root_path`：

```python
APP = FastAPI(root_path="/github-walkthrough-assistant")
```

并确保前端资源引用不再写死为根路径 `/static/...`。

也可以在 Nginx 中额外转发 `/static/`，但不推荐，因为多个项目共用一个 IP 时容易冲突。

### 经验

页面能打开不代表前端部署完整。

如果页面没有样式，优先检查浏览器 Network 面板：

```text
CSS 是否 404
JS 是否 404
资源路径是否从 /static 错误地请求
是否应该带上 /项目路径/static
```


## 7.2 FastAPI 只返回 HTML，但 CSS / JS 没有挂载

### 现象

`github-walkthrough-assistant` 部署后，公网可以打开页面，但页面只有浏览器默认样式，像一个“裸 HTML 骨架”。

进入服务器检查前端目录：

```bash
cd /root/github-walkthrough-assistant/frontend
ls
```

可以看到前端文件都在：

```text
app.js
index.html
styles.css
```

继续查看 `index.html`，发现页面引用的是：

```html
<link rel="stylesheet" href="/styles.css">
<script src="/app.js"></script>
```

但是实际访问静态资源时返回 404：

```bash
curl -I http://127.0.0.1:8010/styles.css
curl -I http://127.0.0.1:8010/app.js
```

这说明：

```text
GET /        能返回 index.html
GET /styles.css 返回 404
GET /app.js     返回 404
```

### 根因

这次问题的核心不是 Nginx 或 systemd 失败，而是 FastAPI 应用层没有暴露静态资源路由。

浏览器打开网页时，不是一次性拿到完整页面，而是分步骤请求：

```text
第一步：请求 /，拿到 index.html
第二步：浏览器解析 index.html，发现还需要 /styles.css 和 /app.js
第三步：继续请求 /styles.css 和 /app.js
```

当前 FastAPI 只配置了：

```text
访问 / → 返回 frontend/index.html
```

但没有配置：

```text
访问 /styles.css → 返回 frontend/styles.css
访问 /app.js     → 返回 frontend/app.js
```

所以浏览器虽然拿到了页面骨架，但拿不到 CSS 和 JS，最终页面就没有样式和交互。

大白话理解：

```text
服务器文件夹里有 styles.css 和 app.js
≠
浏览器可以通过 URL 访问到 styles.css 和 app.js
```

中间必须有一层映射关系，告诉 FastAPI：

```text
浏览器请求某个 URL 时，应该去服务器哪个文件夹里取对应文件
```

这层映射就是“静态资源服务”。

### 解决方式

推荐使用 FastAPI 标准静态文件挂载，把 `frontend` 目录统一挂到 `/static`：

```text
/static/styles.css → frontend/styles.css
/static/app.js     → frontend/app.js
```

然后同步修改 `index.html` 中的引用路径：

```html
<link rel="stylesheet" href="/static/styles.css">
<script src="/static/app.js" defer></script>
```

修复后的访问关系变成：

```text
/                    → 返回 frontend/index.html
/static/styles.css   → 返回 frontend/styles.css
/static/app.js       → 返回 frontend/app.js
```

不推荐为了极限最小改动而分别给 `/styles.css`、`/app.js` 单独写路由。这个方案虽然能临时解决问题，但扩展性较差。后续如果新增图片、字体、图标、更多 JS 文件，还要继续手动补路由。

更好的方式是统一约定：

```text
所有静态资源都走 /static/...
```

这样后续新增资源也能复用同一套规则。

### 验证方式

代码更新到服务器后：

```bash
cd /root/github-walkthrough-assistant
git pull
sudo systemctl restart github-walkthrough-assistant
```

然后验证静态资源是否能被 FastAPI 返回：

```bash
curl -I http://127.0.0.1:8010/static/styles.css
curl -I http://127.0.0.1:8010/static/app.js
```

预期结果：

```text
HTTP/1.1 200 OK
```

如果公网路径是子路径部署，还要继续验证：

```bash
curl -I http://服务器IP/github-walkthrough-assistant/static/styles.css
curl -I http://服务器IP/github-walkthrough-assistant/static/app.js
```

浏览器侧需要强制刷新：

```text
Ctrl + F5
```

避免旧的 CSS / JS 路径被缓存。

### 经验

这次问题说明，云端部署时不能只验证首页是否能打开。

完整网页至少要验证三类请求：

```text
首页 HTML 是否返回 200
CSS / JS 静态资源是否返回 200
后端 API 请求是否返回 200
```

页面能打开，只说明 `index.html` 返回成功；页面是否“完整可用”，还要看 CSS、JS、图片、字体、接口等后续请求是否都能正常加载。

对于 FastAPI 项目，如果由后端返回前端页面，就必须明确区分：

```text
首页路由：返回 index.html
静态资源路由：返回 CSS / JS / 图片等现成文件
API 路由：处理业务请求
```

这也是从“本地能跑”到“云端可用”之间非常典型的工程化差异。


## 7.3 API 路径 404 与长任务 504：从“找不到接口”到“后端处理超时”

### 现象

`github-walkthrough-assistant` 在本地服务器运行正常，但云端公网访问时出现两类问题：

```text
第一阶段：
页面能打开，但点击“分析”按钮后，浏览器 Console 显示：
/api/analyze 404 Not Found

第二阶段：
修复路径后，不再 404，但出现：
504 Gateway Timeout
Unexpected token '<', "<html> <h"... is not valid JSON
```

其中 `Unexpected token '<'` 的含义是：

```text
前端代码以为后端会返回 JSON，于是调用 response.json()
但实际收到的是 Nginx 返回的 HTML 错误页
HTML 以 <html> 开头，所以 JSON 解析失败
```

这说明问题已经不再是单纯的前端解析错误，而是后端请求链路中先发生了 504。

### 排查思路

由于本地服务器可以正常运行，因此优先怀疑云端部署比本地多出来的链路：

```text
浏览器
↓
公网 IP
↓
Nginx 子路径转发
↓
FastAPI / Uvicorn
↓
GitHub API
↓
DeepSeek API
```

排查顺序不是随机尝试，而是沿着请求链路分层验证：

```text
1. 先确认前端是否把请求发到了正确路径
2. 再确认 Nginx 是否把请求转给 8010 后端
3. 再确认 FastAPI 是否接到了请求
4. 再确认 GitHub API / DeepSeek API 是否可访问
5. 最后拆解完整 analyze 流程，定位耗时瓶颈
```

### 问题一：前端 fetch 使用根路径，导致子路径部署下 API 404

公网项目实际访问路径是：

```text
http://8.133.178.210/github-walkthrough-assistant/
```

但前端如果写成：

```javascript
fetch("/api/analyze")
```

浏览器会请求：

```text
http://8.133.178.210/api/analyze
```

也就是说，请求会从网站根路径 `/` 开始找，丢失了：

```text
/github-walkthrough-assistant/
```

这个项目前缀。

因此，正确方式是改成相对路径：

```javascript
fetch("api/analyze")
```

当页面位于：

```text
/github-walkthrough-assistant/
```

时，相对路径会自动解析为：

```text
/github-walkthrough-assistant/api/analyze
```

这和之前 CSS / JS 404 是同类问题：

```html
<!-- 容易在子路径部署下出错 -->
<link rel="stylesheet" href="/styles.css">
<script src="/app.js"></script>
```

子路径部署时，前端代码中要谨慎使用以 `/` 开头的路径。`/api/analyze`、`/static/app.js`、`/styles.css` 都表示“从网站根目录开始找”，而不是“从当前项目路径开始找”。

### 验证方式

检查前端是否仍有根路径写死：

```bash
cd /root/github-walkthrough-assistant
grep -R 'href="/\|src="/\|fetch("/' -n frontend src templates . 2>/dev/null
```

检查公网实际加载到的 JS 是否已经更新：

```bash
curl -s http://8.133.178.210/github-walkthrough-assistant/static/app.js \
  | grep -n "fetch\|api/analyze\|api/plain-explain"
```

浏览器侧验证：

```text
F12 → Network → Disable cache → Ctrl + F5
点击“分析”
观察请求路径是否为：
/github-walkthrough-assistant/api/analyze

而不是：
/api/analyze
```

### 问题二：路径修复后出现 504 Gateway Timeout

路径修复后，请求可以进入后端，但浏览器出现：

```text
504 Gateway Timeout
```

这说明：

```text
Nginx 已经把请求转发给 FastAPI
但 FastAPI 后端长时间没有返回结果
Nginx 等待超时后返回 504
```

此时问题已经从“找不到接口”变成“后端处理时间太长”。

### 完整链路拆解计时

为了确认是不是外部 API 或业务流程耗时，先分别验证：

```text
GitHub API：正常
README 接口：正常
DeepSeek 最小调用：正常，约 2 秒返回
```

然后对 `/api/analyze` 的完整流程做分步计时：

```text
readme 获取：1.73 秒
仓库上下文获取：10.03 秒
README 翻译：82.84 秒
项目概览生成：15.32 秒
总耗时：109.92 秒
```

结论：

```text
真正耗时的不是 GitHub，也不是 DeepSeek 完全不可用，
而是完整 analyze 流程中的 README 翻译步骤太重。
```

本次测试中 README 长度约：

```text
44252 chars
```

其中：

```text
translate_readme(readme.content)
```

单步耗时约 82.84 秒。

### 根因

`/api/analyze` 当前是同步长任务，串行执行：

```text
读取 README
↓
读取仓库上下文
↓
翻译 README
↓
生成项目概览
↓
一次性返回结果
```

完整执行时间约 110 秒。云端公网访问时，中间多了一层 Nginx。Nginx 默认不会无限等待后端响应，因此提前返回 504。

大白话理解：

```text
之前 404：前台没找到这个窗口
现在 504：前台已经把单子递给后厨，但后厨做太久，前台等超时了
```

### Nginx 超时配置修复

先定位真实生效的 Nginx 配置：

```bash
sudo grep -R "127.0.0.1:8010" -n /etc/nginx
```

本次定位到：

```text
/etc/nginx/sites-available/ecno-assist
/etc/nginx/sites-enabled/ecno-assist
```

通常 `sites-enabled` 是软链接，优先编辑源文件：

```bash
sudo nano /etc/nginx/sites-available/ecno-assist
```

在当前项目对应的 location 中增加 timeout：

```nginx
location /github-walkthrough-assistant/ {
    proxy_pass http://127.0.0.1:8010/;
    proxy_http_version 1.1;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;

    proxy_connect_timeout 60s;
    proxy_send_timeout 300s;
    proxy_read_timeout 300s;
    send_timeout 300s;
}
```

修改后检查并重载：

```bash
sudo nginx -t
sudo systemctl reload nginx
```

由于完整任务约 110 秒，公网 API 测试时要给足等待时间：

```bash
curl -i --max-time 180 -X POST "http://8.133.178.210/github-walkthrough-assistant/api/analyze" \
  -H "Content-Type: application/json" \
  -d '{"repo_url":"https://github.com/Hmbown/CodeWhale"}'
```

### 为什么不能只看浏览器报错

浏览器最终只会显示：

```text
分析失败
504
Unexpected token '<'
```

但它不直接告诉你是哪一层出问题。

更好的方式是：

```text
浏览器负责发真实公网请求
SSH 里用 journalctl 实时看后端日志
```

命令：

```bash
sudo journalctl -u github-walkthrough-assistant -f -l
```

推荐方式：

```text
一个入口发请求
另一个窗口看日志
```

不建议浏览器和 curl 同时发起完整分析请求，因为每次都会触发一次约 110 秒的 DeepSeek 调用，既消耗额度，也会让日志混杂。

### 前端错误处理优化

前端不应默认所有响应都是 JSON。Nginx 返回 504 时，响应是 HTML，不是 JSON。建议先读取文本，再判断状态码：

```javascript
const response = await fetch("api/analyze", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ repo_url: repoUrl }),
});

const text = await response.text();

if (!response.ok) {
  throw new Error(`请求失败：HTTP ${response.status}，${text.slice(0, 200)}`);
}

const data = JSON.parse(text);
```

这样即使后端返回 504 HTML，也能显示更明确的错误，而不是出现难以理解的：

```text
Unexpected token '<'
```

### 经验

这次问题说明，云端和本地最大的差异不只是“能不能运行代码”，而是多了公网代理层和超时机制。

核心经验：

```text
本地正常，不代表云端正常
404 和 504 是两个完全不同层级的问题
子路径部署时前端不能随便写根路径
长任务不适合通过同步 HTTP 请求一次性返回
Nginx timeout 可以缓解，但不是长期架构解法
```

短期解决：

```text
修正 fetch 相对路径
调大 Nginx proxy_read_timeout
前端增强非 JSON 响应处理
```

长期优化：

```text
限制 README 翻译长度
将 README 翻译改为可选功能
/api/analyze 改造成异步任务模式：
POST /api/analyze → 返回 task_id
GET /api/tasks/{task_id} → 轮询任务结果
```

# 8. GitHub 代码同步类问题

## 8.1 GitHub 更新后，服务器如何同步

### 是否需要重启服务

`git pull` 只是把 GitHub 上的新代码同步到服务器文件系统。是否需要重启，取决于这次改了什么：

| 修改内容 | 是否需要重启 / 重载 | 原因 |
|---|---|---|
| 只改 `frontend/index.html`、`styles.css`、`app.js` | 通常不需要重启，浏览器强制刷新即可 | 静态文件是现取现用的 |
| 改 FastAPI / Python 后端代码 | 需要 `systemctl restart` | 运行中的 Python 进程不会自动加载新代码 |
| 改 `requirements.txt` / Python 依赖 | 需要安装依赖后重启 | 新依赖不会自动进入虚拟环境 |
| 改 Nginx 配置 | 需要 `nginx -t` 后 reload | Nginx 配置不会自动生效 |

这次 `/static` 挂载问题涉及 FastAPI 入口代码和 `index.html`，因此推荐流程是：

```bash
cd /root/github-walkthrough-assistant
git pull
sudo systemctl restart github-walkthrough-assistant
curl -I http://127.0.0.1:8010/static/styles.css
curl -I http://127.0.0.1:8010/static/app.js
```

如果只是后续微调 CSS 样式，通常 `git pull` 后浏览器 `Ctrl + F5` 即可。

### 推荐流程

服务器上不要重新 clone，而是在现有项目目录中更新：

```bash
cd /root/economist-content-pipeline
git status
git pull
source .venv/bin/activate
pip install -r requirements.txt
sudo systemctl restart economist-content-pipeline
sudo systemctl status economist-content-pipeline
```

第二个项目类似：

```bash
cd /root/github-walkthrough-assistant
git status
git pull
source .venv/bin/activate
pip install -r requirements.txt
sudo systemctl restart github-walkthrough-assistant
sudo systemctl status github-walkthrough-assistant
```

### 如果服务器本地代码和 GitHub 冲突

如果服务器只是部署环境，不保留本地修改，可以强制以 GitHub 为准：

```bash
git fetch origin
git reset --hard origin/main
```

或：

```bash
git fetch origin
git reset --hard origin/master
```

### 经验

推荐工作流：

```text
本地 / Codex 修改代码
↓
commit
↓
push 到 GitHub
↓
云服务器 git pull
↓
pip install -r requirements.txt
↓
systemctl restart 服务
```

云服务器最好只做部署，不作为主要开发环境。

# 11. 这次部署带来的工程经验

## 11.1 502 不是一个笼统错误，而是一个明确方向

502 的核心含义是：

```text
Nginx 可以收到请求
但后端 upstream 不可用
```

所以排查重点不是重装 Nginx，而是检查：

```text
后端进程是否存在
端口是否监听
systemd 是否 running
后端日志是否报错
```

---

## 11.2 手动运行和生产部署是两回事

手动执行：

```bash
uvicorn ...
```

只能说明代码暂时能跑。

生产部署需要的是：

```text
systemd 托管
开机自启
异常重启
日志可追踪
SSH 断开不影响服务
```

---

## 11.3 部署入口必须从项目结构中推导，而不是猜

同样是 Python Web 项目，启动方式可能完全不同：

```text
FastAPI
Flask
http.server
纯前端静态站点
```

所以部署前必须先确认：

```text
项目入口
框架类型
依赖环境
运行端口
静态资源目录
```

---

## 11.4 Nginx 的本质是入口路由器

Nginx 不负责业务逻辑。它的关键作用是：

```text
接收公网请求
根据域名或路径分流
转发到不同本机端口
处理反向代理请求头
```

在没有域名的情况下，一个公网 IP 可以用多个 location 管理多个项目：

```text
/          → 8000
/project2/ → 8010
/project3/ → 8020
```

---

## 11.5 子路径部署比根路径部署更容易踩坑

根路径部署通常是：

```text
/
```

子路径部署是：

```text
/github-walkthrough-assistant/
```

后者会额外带来：

```text
Nginx 是否剥离路径前缀
FastAPI 是否设置 root_path
前端资源是否写死 /static
浏览器是否请求到了正确 CSS / JS
```

---

## 11.6 云服务器应该是部署环境，不是开发环境

更稳妥的流程是：

```text
本地开发
↓
GitHub 版本管理
↓
服务器 git pull
↓
重启服务
```

这样可以避免服务器上产生不可追踪的临时修改。


## 11.7 长任务要单独考虑代理超时和任务架构

本地直接调用接口时，长任务可以一直等待。但云端公网访问通常会经过 Nginx，Nginx 对后端响应有等待时间限制。

如果一个接口需要几十秒甚至一两分钟才能返回，例如：

```text
读取外部仓库
调用大模型翻译长 README
生成项目分析报告
```

就不能只验证“代码最终能跑完”，还要验证：

```text
Nginx 是否愿意等这么久
浏览器是否能保持请求
前端是否能正确展示等待状态
失败时是否能显示清楚错误
```

短期可以通过调大 Nginx 的 `proxy_read_timeout` 解决公网 504，但更好的长期方案是异步任务化：

```text
前端提交任务
↓
后端立即返回 task_id
↓
后台慢慢执行分析
↓
前端轮询任务状态并展示结果
```

经验是：

```text
同步接口适合秒级响应
分钟级任务应该按后台任务设计
```

---

# 12. 总结

> 真正的云端部署，不是“把代码传到服务器并跑起来”，而是让一个服务在公网环境下可访问、可持续运行、可自动恢复、可定位问题。  
> Nginx 管入口，systemd 管进程，GitHub 管版本，虚拟环境管依赖；只有这几层全部对齐，项目才算从本地 Demo 变成了可部署的 Web 服务。
